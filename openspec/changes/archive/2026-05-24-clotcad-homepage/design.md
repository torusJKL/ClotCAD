## Context

ClotCAD is a parametric CAD application with a Qt6/OCCT 3D viewer and Lisp frontend. It currently has no public documentation website. The project uses OpenSpec workflow for change management. The existing CI pipeline (`release.yml`) already builds OCCT (cached), compiles `libclotcad.so`, and packages AppImage/tarball distributions on tag pushes.

Staple is a Common Lisp documentation generator that produces static HTML from docstrings, README files, and other documents. It uses Clip for templating and has a Markdown extension (`staple-markdown`) for `.md` source support. Staple Server provides a live preview during development.

## Goals / Non-Goals

**Goals:**
- Static documentation site generated by Staple, deployed to GitHub Pages
- Versioned output: `/latest/` and `/v<semver>/` per release
- API reference auto-generated from docstrings in the `:clotcad`, `:clotcad-user`, and `:cl-occt` packages
- Custom Clip template matching ClotCAD's light-mode Fluent-inspired visual style
- PDF cheatsheet compiled from Typst, linked from navigation
- CI automation on tag push: generate docs, compile cheatsheet, deploy to `gh-pages` branch
- DNS: `clotcad.com` pointed to GitHub Pages

**Non-Goals:**
- Hand-written API reference pages (AI-oriented `clotcad-api.md` excluded from Staple pipeline)
- Dark mode on the homepage (the template is light-mode only; dark mode is not a requirement)
- Multi-language documentation
- Dynamic server-side features (everything is static HTML)

## Decisions

1. **Staple over mkdocs/sphinx/static-site-gen**
   - Staple reads Lisp source directly — no separate docstring extraction step
   - Automatically generates definitions index from exported symbols
   - Cross-references symbols across packages (links `show-grid` to its definition)
   - Only available CL-native doc tool that handles ASDF systems directly

2. **`staple.ext.lisp` in root with delegation to `homepage/`**
   - Staple hardcodes the extension file as `staple.ext.lisp` in the system source root
   - Keeping it tiny (`(load "homepage/setup.lisp")`) keeps root clutter minimal
   - All real customization lives in `homepage/setup.lisp`

3. **Tags-only trigger over push-to-main**
   - Users browsing `clotcad.com` should see docs matching their installed version
   - Avoids docs drifting ahead of releases

4. **Versioned output with `latest/` alias**
   - `gh-pages` branch root has `index.html` → `<meta refresh>` to `latest/`
   - CI extracts version from `v0.2.2` tag, publishes to both `/v0.2.2/` and `/latest/`
   - Old versions remain accessible at their permanent path

5. **CI reuse of existing OCCT cache**
   - The existing `release.yml` already caches `.local/` under key `occt-8.0.0-${{ runner.os }}-${{ hashFiles('CMakeLists.txt') }}`
   - Homepage workflow uses the same cache key — OCCT is rarely rebuilt
   - `just viewer` step is needed to produce `libclotcad.so` (required by CFFI at load time)

6. **Markdown docstrings with cross-references, not syntax highlighting**
   - Staple's `markup-code-snippets` converts symbols in code blocks to clickable xref links
   - No JS syntax highlighter — CDN dependency avoided; cross-referencing provides sufficient navigation

## Risks / Trade-offs

- **Staple load-order sensitivity**: Staple's recording mechanism works best when the system is loaded *by* Staple, not pre-loaded. Mitigation: We explicitly set `:packages` in setup.lisp, bypassing the recording mechanism entirely.
- **CI build time**: Even with OCST cached, SBCL+Quicklisp setup and Staple's dependency download takes ~2-3 minutes per run. Acceptable for tag-triggered workflow (infrequent).
- **`cl-occt` dependency**: Staple must load `:cl-occt` to document its exported symbols. If `cl-occt` has unresolved dependencies or build issues in CI, docs generation fails. Mitigation: same build environment as `release.yml` is already proven.
- **No dark mode**: The homepage template is light-only. Adding dark mode later would require editing the Clip template and style.css. Intentional non-goal for this change.
